## 4. Defects Found

### Defect 1

Type: Functional / Integration Defect  
Severity: High  
Title: Preview attendance `Clock in` and `Refresh entry` paths fail backend validation

Steps to reproduce:

1. Sign in to the application with a seeded user that can access `/preview`.
2. Open `/preview` and navigate to the `Attendance` module.
3. Trigger the primary attendance action in a state that should create a fresh same-day entry or refresh a completed same-day entry.
4. Observe the attendance request body generated by the preview flow and the API response.

Expected result:

- the primary attendance action should store successfully, show a centered success popup, and refresh the attendance review list

Actual result:

- the preview flow builds a backend request with `timeOut: null`
- the attendance API validation schema accepts `timeOut` only as an omitted field or ISO datetime string
- the preview-equivalent runtime request returns HTTP `400` with `DEBUG-API-ATTENDANCE-VALIDATION`

Evidence:

- preview request body sets `timeOut: isClockOut ? actionTimestamp : null`
- attendance validation schema defines `timeOut` as `z.iso.datetime().optional()`
- runtime retest produced:
  - preview-equivalent fresh attendance request -> HTTP `400`
  - debug ref -> `DEBUG-API-ATTENDANCE-VALIDATION`

Impact:

- the headline fix in this slice is only partially working
- `Clock out` and some flagged-update paths can succeed, but `Clock in` and `Refresh entry` remain broken in backend mode
- this blocks QA approval of the attendance preview workflow
